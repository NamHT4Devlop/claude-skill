🧭 1. How it works (1-minute read)

Just remember 3 separate layers:

LayerWhat it isWhere it lives
🔧 Toolskills / commands / agents~/.claude/ — install once, used by every project
📚 KB (Knowledge Base)knowledge about one specific project<project>/knowledge-base/ — one copy per project, local
🗑️ Artifactplans, code, evidence generated on each run<project>/spec-kit-sessions/ — ignored by git machine-wide
💡 The mantra

Tool is global · KB is per project · one project per session. What decides "which project you're working on" is the folder you open Claude Code in (cwd) — not where the tool is installed.

🛠️ 2. Setup steps

Already done on the current machine. Use these steps when installing on a new machine, or to re-verify.

1

Get the tool's source

Clone the repo to a fixed location (already at /Users/MAC/AI-TOOL/nam-claude-skill on this machine).

# SSH (recommended if your GitHub uses an SSH key)
git clone git@github.com:NamHT4Devlop/nam-claude-skill.git ~/nam-claude-skill

Below this folder is referred to as <TOOL>. On this machine <TOOL> = /Users/MAC/AI-TOOL/nam-claude-skill.

2

Install the tool for yourself (symlink into ~/.claude)

A single command — it symlinks the skills + agents + commands into your home folder. Commands are prefixed with namht- so they don't shadow built-ins (like /help).

/Users/MAC/AI-TOOL/nam-claude-skill/scripts/personal-install.sh

Because it's symlinked → a later git pull in <TOOL> updates automatically, no reinstall.

3

Prevent leaks — global gitignore (machine config, touches no repo)

So even if you accidentally run a command in a team repo, the generated KB & sessions stay local + never committed.

touch ~/.gitignore_global
printf '%s\n' 'spec-kit-sessions/' 'knowledge-base/' 'CLAUDE.local.md' '.spec-kit/' >> ~/.gitignore_global
git config --global core.excludesfile ~/.gitignore_global
4

Verify

Confirm the symlinks & gitignore work:

# see the namht-* symlinks pointing back to <TOOL>
ls -la ~/.claude/skills | grep namht-
# confirm KB/sessions are ignored in every repo
git config --global core.excludesfile

Then open Claude Code, type / and check that /namht-build, /namht-ask… show up.

⚠️ Pick only one install method

If you used personal-install.sh, do NOT also run /plugin install for the same plugin — you'll get duplicate commands (defined twice).

▶️ 3. Daily usage

Important: always open Claude Code inside the project you want to work on, so it reads that project's KB.

# 1) cd into the right project
cd /Users/MAC/AI-TOOL/human-essentials
claude

# 2) first time on a project: create the Knowledge Base
/namht-scan

# 3) then use freely
/namht-ask   how does the payment flow work?
/namht-build add a forgot-password feature via email OTP, expiring after 10 minutes
/namht-review
💬 Tip

You don't have to type a slash command — a skill auto-activates when you speak naturally, e.g. "review this file for me" or "build feature X per the KB".

❓ 8. FAQ

Does each project need a .claude folder?

No. You install once in ~/.claude (global) → every project can use it. Only create .claude inside a repo when you intentionally want to ship the tool alongside the repo for the team — which you don't.

Can I reuse an old KB generated on another machine?

100%. The KB is plain Markdown in knowledge-base/; just copy it into the right project folder and every command reads it instantly — no regeneration.

For a plain-skill install, do I need to copy the resources/ folder?

No. Each skill bundles what it needs in its own references/. The root resources/ is just the canonical source for the plugin form.

What about a monorepo?

Opposite of a multi-repo workspace: open Claude Code at the repo root, put the KB at the root with knowledge-base/modules/ (the scan command generates per-module docs automatically).
