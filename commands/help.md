Show the user a concise guide to the **Spec Kit** plugin. Then check whether the current repo
has a `knowledge-base/` folder (Glob for `knowledge-base/*.md`) and report its status
(present + how many docs, or missing) with a suggested next step.

Commands (all namespaced under `/spec-kit:`):

| Command | What it does |
|---------|--------------|
| `/spec-kit:scan` | Generate the Knowledge Base from the codebase (16 docs + review-skills + per-module). Run this first on a new repo. |
| `/spec-kit:rescan` | Update the KB incrementally after code changes (git-diff aware). |
| `/spec-kit:build <requirement>` | Full 13-step pipeline: clarify → plan → code → review → test → evidence → update KB. |
| `/spec-kit:review [file]` | Two-phase review: quality checklist + business consistency vs the KB. |
| `/spec-kit:ask <question>` | Q&A grounded in the KB — plain language + Mermaid diagram + technical detail. |
| `/spec-kit:plan <epic>` | PO/BA: Epic → features → impact → user stories (Given/When/Then) → sprint plan. |
| `/spec-kit:map [scope]` | Dependency graph (imports/DI/calls/inheritance) → Mermaid, enriched with business meaning. |
| `/spec-kit:document <topic>` | Business↔code field-level technical document for a feature/entity/module. |

Recommended flow: **scan** once → **ask/map/document** to understand → **plan** to break down
work → **build** to implement → **rescan** to keep the KB fresh. The KB lives in
`knowledge-base/` and grounds every other command — KBs generated by the original Auto Spec
Kit VS Code extension are reused as-is.
